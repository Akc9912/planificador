# 1.3 Mapear Entidades JPA

## Objetivo

Modelar y validar las entidades JPA nuevas sobre el esquema objetivo usando UUID, enums string y tipos compatibles con PostgreSQL, sin importar relaciones directas con `auth.users` ni arrastrar decisiones legacy al codigo modular nuevo.

## Por que se hace

La migracion del backend depende de que las entidades nuevas hablen el idioma real de la base de datos. Si se sigue modelando con enteros o relaciones falsas, la deuda tecnica se traslada modulo por modulo.

## Pasos de ejecucion

1. Revisar modelos ya migrados y el schema fuente en `humanis_db_init.sql`.
2. Confirmar que los ids nuevos usan `UUID` y no `Integer`.
3. Confirmar que los enums usan `@Enumerated(EnumType.STRING)`.
4. Confirmar que los arrays y colecciones complejas tienen estrategia de mapeo definida.
5. Confirmar que `auth.users` se trata como referencia externa por `user_id`, no como entidad propia.
6. Compilar para validar imports, anotaciones y tipos.

## Comandos exactos

```bash
find src/main/java/aktech/planificador/modules -path "*/model/*.java" | sort
grep -R "UUID\|@Enumerated\|@Entity" -n src/main/java/aktech/planificador | head -n 40
grep -n "CREATE TABLE\|uuid\|UUID" humanis_db_init.sql | head -n 60
./mvnw -q -DskipTests compile
```

## Ejemplo de codigo

```java
@Entity
public class Career {

    @Id
    private UUID id;

    @Enumerated(EnumType.STRING)
    private CareerStatus status;

    @Column(name = "user_id", nullable = false)
    private UUID userId;
}
```

## Verificacion

- Los modelos nuevos usan UUID.
- Los enums nuevos se persisten como string.
- El `user_id` se maneja como valor simple cuando refiere a `auth.users`.
- El schema SQL y las entidades no se contradicen en nombres o tipos base.
- El proyecto compila sin errores de anotaciones JPA.

## Resultado esperado

Las entidades nuevas quedan alineadas al schema objetivo y listas para evolucionar modulo por modulo sin refactor global previo.

## Criterio de cierre

- [ ] No se usan ids enteros en codigo nuevo.
- [ ] Los enums del codigo nuevo usan persistencia string.
- [ ] La referencia a usuario externo se resuelve con `user_id`.
- [ ] El schema SQL se toma como fuente de verdad.
- [ ] El build compila en verde.

## Troubleshooting breve

Si aparece una duda de mapeo, priorizar el schema de `humanis_db_init.sql` por sobre el modelo legacy. No introducir compatibilidades temporales que mezclen ambos enfoques dentro del modulo nuevo.
