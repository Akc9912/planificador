# 1.3 Mapear Entidades JPA

## Objetivo

Validar la base JPA del backend modular activo sobre el esquema objetivo usando UUID, enums string y tipos compatibles con PostgreSQL, sin importar relaciones directas con `auth.users` ni arrastrar decisiones legacy al codigo modular nuevo.

## Por que se hace

La migracion del backend depende de que las entidades nuevas hablen el idioma real de la base de datos. Si se sigue modelando con enteros o relaciones falsas, la deuda tecnica se traslada modulo por modulo.

## Pasos de ejecucion

1. Revisar modelos ya migrados del backend modular activo y el schema fuente en `humanis_db_init.sql`.
2. Confirmar que los ids nuevos usan `UUID` y no `Integer`.
3. Confirmar que los enums usan `@Enumerated(EnumType.STRING)`.
4. Confirmar que los arrays y colecciones complejas tienen estrategia de mapeo definida.
5. Confirmar que `auth.users` se trata como referencia externa por `user_id`, no como entidad propia.
6. Dejar explicitado el borde de alcance: Subject y Equivalence completos se cierran en Fase 2.
7. Compilar para validar imports, anotaciones y tipos.

## Ejemplo de codigo

```java
@Entity
public class Career {

    @Id
    private UUID id;

    @Enumerated(EnumType.STRING)
    private CareerStatus status;

    @Column(name = "user_id", nullable = false)
    private UUID userId;
}
```
