# 1.7 Testing

## Objetivo

Definir y ejecutar la suite minima de pruebas para la base modular, cubriendo auth, career y reglas arquitectonicas sin agregar nuevas pruebas al codigo legacy.

## Por que se hace

El proyecto necesita un piso tecnico de confianza antes de avanzar a Subject y Equivalence. Sin ese piso, cada paso posterior se apoya en supuestos no verificados.

## Pasos de ejecucion

1. Ejecutar la suite modular actual.
2. Confirmar que los tests cubren auth, career y guardrails.
3. Revisar que no se hayan agregado pruebas nuevas al codigo legacy.
4. Documentar resultados y dejar la orden de ejecucion lista para CI o reproduccion local.

## Comandos exactos

```bash
./mvnw -q -DskipTests=false -Dtest=ModuleBoundariesTest,CareerServiceTest,CareerControllerTest,AuthSessionServiceTest,AuthJwtAuthenticationFilterTest,AuthModuleControllerTest,SecurityConfigAuthIntegrationTest test
./mvnw -q -DskipTests compile
```

## Ejemplo de codigo

```java
@Test
void modulesShouldNotImportLegacyLayers() throws IOException {
    assertTrue(violations.isEmpty());
}
```

## Verificacion

- La suite definida corre en verde.
- Los tests cubren los modulos migrados y la regla arquitectonica.
- No se agregaron tests nuevos en capas legacy.
- El build sigue compilando despues de ejecutar pruebas.

## Resultado esperado

Existe una base minima de regression testing que acompana el cierre de Fase 1 y sirve como referencia de calidad para el resto de la migracion.

## Criterio de cierre

- [ ] La suite modular minima corre localmente.
- [ ] Los tests reflejan solo modulos migrados y guardrails.
- [ ] No se agregan tests nuevos al legacy.
- [ ] La orden de ejecucion queda documentada para repetirse.

## Troubleshooting breve

Si Maven muestra verde pero los tests no ejecutan, revisar la propiedad `skipTests`. Un falso verde en esta fase invalida toda la evidencia tecnica posterior.
