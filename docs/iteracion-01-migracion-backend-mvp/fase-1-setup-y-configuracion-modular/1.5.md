# 1.5 APIs REST Basicas

## Objetivo

Dejar definido y verificable el contrato REST base del backend modular activo hoy, con autenticacion, ownership, DTOs de entrada/salida y manejo consistente de errores para los modulos ya activos.

## Convencion de rutas vigente

- Career usa prefix `/careers`.
- Auth modular usa prefix `/auth`.
- Seguridad y pruebas actuales ya estan alineadas con esta convencion.

## Patron oficial de autenticacion y ownership

- El principal autenticado para requests JWT validos se guarda como userId en formato string UUID.
- En controladores de modulos activos, el userId se extrae desde `authentication.getName()` y se parsea a UUID.
- La identidad no debe viajar por query ni path cuando el endpoint ya requiere autenticacion.
- Decision de fase: no crear helper compartido para esta extraccion durante Fase 1. Se conserva el patron actual y se evalua refactor en Fase 2.

## Patron base de errores vigente en Fase 1

- Decision explicita de fase: no introducir todavia un envelope JSON global para todos los errores.
- Las excepciones manejadas por `GlobalExceptionHandler` responden hoy body de texto plano con el mensaje correspondiente.
- Esto aplica hoy a `400`, `401`, `403`, `404`, `409` y `500` resueltos por el handler global.
- Los endpoints de auth delegados a Supabase (`/auth/login`, `/auth/register`, `/auth/register-admin`, `/auth/change-password`) responden `410 GONE` con body JSON minimo: `message` + `operation`.
- Un formato unificado tipo `status/error/message/path/timestamp` queda diferido a un refactor transversal posterior, fuera del cierre de Fase 1.

## Por que se hace

Sin un contrato REST base cerrado no existe una interfaz estable para backend ni una base reutilizable para seguir la migracion de modulos.

## Pasos de ejecucion

1. Revisar controladores y DTOs existentes en modulos activos del backend modular (Career/Auth) y endpoints legacy aun activos.
2. Confirmar que cada endpoint sensible exige autenticacion.
3. Verificar que el `userId` se toma del contexto de seguridad y no del request cuando aplique.
4. Levantar la aplicacion para probar endpoints base del contrato activo.
5. Documentar resultado de pruebas para rutas base de Career/Auth y respuestas esperadas.
6. Dejar explicitado que CRUD completo de Subject/Equivalence se cierra en Fase 2.

## Ejemplo de codigo

```java
@GetMapping("/careers")
public List<CareerResponseDto> listByUser() {
    UUID userId = getAuthenticatedUserId();
    return careerService.listByUser(userId);
}
```

## Verificacion

- Existen endpoints base documentados para los modulos activos (Career/Auth).
- Un request autenticado devuelve respuesta valida en al menos una ruta de Career y una de Auth.
- Los endpoints que requieren ownership no reciben `userId` por query o path cuando la identidad debe salir del token.
- El patron de extraccion de userId desde `authentication.getName()` queda documentado como regla vigente.
- Los DTOs separan request y response en los modulos activos.
- Las rutas documentadas respetan la convencion vigente de los controladores (`/careers` y `/auth`).
- El formato de error documentado coincide con el comportamiento actual del backend.
- Queda documentado que Subject/Equivalence completos se cierran en Fase 2.

## Resultado esperado

El backend expone un contrato REST base minimo, autenticado y coherente para los modulos activos, util para cerrar Fase 1 y habilitar Fase 2.
