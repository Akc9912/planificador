# 2.2 Implementar Subject Module

## Objetivo

Construir el modulo `subject` sobre esquema UUID, cubriendo materias, modulos evaluables y horarios semanales, sin importar implementaciones de otros modulos ni reutilizar clases legacy como dependencia directa.

## Por que se hace

Subject concentra gran parte de la logica de negocio del producto. Es el modulo mas complejo y tambien el de mayor riesgo de acoplamiento si se implementa apurado o usando servicios legacy como atajo.

## Pasos de ejecucion

1. Confirmar que Career esta cerrado antes de avanzar.
2. Crear o completar la estructura `modules/subject` con capas estandar.
3. Modelar `subjects`, `subject_modules` y `subject_schedules` con UUID.
4. Resolver dependencias con Career solo por `CareerApi` y eventos compartidos.
5. Exponer endpoints CRUD y secundarios del modulo.
6. Agregar pruebas del modulo sin tocar codigo legacy.

## Comandos exactos

```bash
find src/main/java/aktech/planificador/modules/subject -type f | sort
grep -R "CareerApi\|SubjectApi\|CareerDeletedEvent" -n src/main/java/aktech/planificador/modules/subject src/main/java/aktech/planificador/shared
./mvnw -q -DskipTests compile
./mvnw -q -DskipTests=false -Dtest=Subject*Test,ModuleBoundariesTest test
```

## Ejemplo de codigo

```java
@Service
public class SubjectService {

    private final CareerApi careerApi;

    public SubjectService(CareerApi careerApi) {
        this.careerApi = careerApi;
    }

    public void validateCareerOwnership(UUID userId, UUID careerId) {
        if (!careerApi.userOwnsCareer(userId, careerId)) {
            throw new NotFoundException("Career not found for user");
        }
    }
}
```

## Verificacion

- Existe estructura modular completa para Subject.
- El modulo no importa `CareerService` ni capas legacy.
- Los ids nuevos son UUID.
- Existen endpoints para materias, modulos evaluables y horarios.
- Hay pruebas propias del modulo y `ModuleBoundariesTest` sigue en verde.

## Resultado esperado

Subject queda preparado como modulo central del MVP, desacoplado y compatible con las reglas de migracion incremental por modulo.

## Criterio de cierre

- [ ] La estructura del modulo existe y compila.
- [ ] Subject usa contratos compartidos para dependencias cruzadas.
- [ ] Se implementan CRUD, schedules y subject modules.
- [ ] No hay imports directos a legacy ni a otros modulos.
- [ ] Las pruebas del modulo y el guardrail pasan.

## Troubleshooting breve

Si faltan pruebas, crearlas dentro del modulo nuevo. No usar tests del legacy como sustituto ni como excusa para saltear cobertura minima.
