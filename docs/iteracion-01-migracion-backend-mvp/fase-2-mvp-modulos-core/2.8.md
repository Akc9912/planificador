# 2.8 Busqueda y Filtros

## Objetivo

Agregar capacidades de busqueda, filtro y orden minimo en los endpoints del MVP para que el backend entregue datos utiles sin exigir filtrado pesado en frontend.

## Por que se hace

Un CRUD sin filtros obliga al cliente a descargar y procesar de mas. Esta tarea baja la carga del frontend y mejora la experiencia de uso del backend desde el primer consumo real.

## Pasos de ejecucion

1. Definir filtros minimos por modulo core.
2. Implementar consultas por nombre, codigo, estado, ano y semestre donde aplique.
3. Agregar ordenamientos y parametros claros.
4. Documentar el contrato de filtros.
5. Agregar pruebas de consulta.

## Comandos exactos

```bash
grep -R "Pageable\|Sort\|findBy\|search\|filter" -n src/main/java/aktech/planificador/modules
./mvnw -q -DskipTests compile
./mvnw -q -DskipTests=false -Dtest=CareerControllerTest,Subject*Test test
```

## Ejemplo de codigo

```java
Page<Subject> page = subjectRepository.findByCareerIdAndStatus(careerId, status, pageable);
```

## Verificacion

- Los endpoints core aceptan filtros utiles.
- La consulta no obliga a traer todo para filtrar del lado cliente.
- El contrato de parametros queda claro.

## Resultado esperado

El backend expone listados mas utiles y preparados para pantallas reales, sin sobredescarga de datos innecesarios.

## Criterio de cierre

- [ ] Existen filtros minimos documentados.
- [ ] Existen consultas o especificaciones para soportarlos.
- [ ] Hay pruebas de filtro u orden.
- [ ] El build sigue en verde.

## Troubleshooting breve

No empezar con filtros complejos si aun no existe el listado base correcto. Primero asegurar exactitud, luego optimizar variedad de consultas.
