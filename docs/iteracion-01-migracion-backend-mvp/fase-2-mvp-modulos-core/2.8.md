# 2.8 Busqueda y Filtros

## Objetivo

Agregar capacidades de busqueda, filtro y orden minimo en los endpoints del MVP para que el backend entregue datos utiles sin exigir filtrado pesado en frontend.

## Por que se hace

Un CRUD sin filtros obliga al cliente a descargar y procesar de mas. Esta tarea baja la carga del frontend y mejora la experiencia de uso del backend desde el primer consumo real.

## Pasos de ejecucion

1. Definir filtros minimos por modulo core.
2. Implementar consultas por nombre, codigo, estado, ano y semestre donde aplique.
3. Agregar ordenamientos y parametros claros.
4. Documentar el contrato de filtros.
5. Agregar pruebas de consulta.

## Comandos exactos

```bash
grep -R "searchByCareerWithFilters\|resolveSort\|/career/{careerId}/search" -n src/main/java/aktech/planificador/modules/subject
./mvnw -q -DskipTests compile
```

## Ejemplo de codigo

```java
Sort sort = Sort.by(Sort.Direction.DESC, "year");
List<Subject> subjects = subjectRepository.searchByCareerWithFilters(careerId, status, name, code, year, semester, sort);
```

## Contrato implementado en Subject

- `GET /subjects/career/{careerId}/search`
- Parametro `name` (opcional): busqueda parcial case-insensitive por nombre.
- Parametro `code` (opcional): busqueda parcial case-insensitive por codigo.
- Parametro `status` (opcional): estado normalizado (`pendiente`, `cursando`, `regular`, `aprobada`, `libre`).
- Parametro `year` (opcional): filtro exacto por anio.
- Parametro `semester` (opcional): filtro exacto por semestre.
- Parametro `sortBy` (opcional): `name`, `code`, `status`, `year`, `semester`, `credits`, `hours`, `createdAt`, `updatedAt`.
- Parametro `sortDir` (opcional): `asc` o `desc`.
- Orden por defecto: `name asc`.
- Fuente de verdad: `SubjectService#listByCareerWithFilters` + `SubjectRepository#searchByCareerWithFilters`.
