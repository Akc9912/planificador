# 2.1 Implementar Career Module

## Objetivo

Cerrar el modulo `career` como primer modulo completo del MVP, incluyendo CRUD, ownership, endpoint de metricas para admin, contrato compartido y pruebas de servicio/controlador.

## Por que se hace

Career es el modulo con mayor avance real y el template tecnico para el resto. Si no queda cerrado primero, Subject y Equivalence se construyen sobre una base todavia inestable.

## Pasos de ejecucion

1. Auditar la carpeta `modules/career` y confirmar sus capas.
2. Verificar que el modulo implementa ownership con `userId` desde el token.
3. Verificar que expone contratos compartidos para otros modulos.
4. Verificar que publica eventos cuando corresponde.
5. Ejecutar las pruebas del modulo y el guardrail de arquitectura.
6. Cerrar cualquier hueco pendiente antes de abrir Subject.

## Comandos exactos

```bash
find src/main/java/aktech/planificador/modules/career -type f | sort
find src/test/java/aktech/planificador/modules/career -type f | sort
grep -R "implements CareerApi\|CareerDeletedEvent\|userId" -n src/main/java/aktech/planificador/modules/career src/main/java/aktech/planificador/shared
./mvnw -q -DskipTests=false -Dtest=CareerServiceTest,CareerControllerTest,ModuleBoundariesTest test
```

## Ejemplo de codigo

```java
@Service
public class CareerService implements CareerApi {

    public List<CareerResponseDto> getCareersByUser(UUID userId) {
        return careerRepository.findByUserId(userId)
                .stream()
                .map(careerMapper::toResponse)
                .toList();
    }
}
```

## Verificacion

- El modulo tiene `controller`, `service`, `repository`, `model` y `dto`.
- Los endpoints no reciben `userId` cuando debe salir del token.
- Existen pruebas de servicio y controlador.
- `CareerService` sirve como contrato para otros modulos via `CareerApi`.

## Resultado esperado

Career queda cerrado como modulo estable, probado y reutilizable, y se convierte en referencia para la migracion de Subject y Equivalence.

## Criterio de cierre

- [ ] CRUD y ownership funcionan.
- [ ] Existe contrato compartido para consultas cruzadas.
- [ ] Se publica evento de dominio al eliminar carrera, si aplica.
- [ ] Las pruebas de Career y arquitectura quedan en verde.
- [ ] No queda ningun acoplamiento nuevo con codigo legacy.

## Troubleshooting breve

No abrir trabajo real de Subject mientras Career tenga huecos funcionales o tests rojos. La regla operativa actual exige cerrar este modulo primero.
