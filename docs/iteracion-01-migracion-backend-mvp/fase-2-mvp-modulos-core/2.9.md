# 2.9 Endpoints de Dashboard

## Objetivo

Exponer endpoints que resuman el estado academico del usuario para la vista principal: recomendaciones, materias activas, estadisticas y alertas utiles.

## Por que se hace

El dashboard concentra el valor visible del backend. Esta tarea convierte calculos y estados internos en un contrato de lectura listo para consumo de interfaz, sin mover logica compleja al frontend.

## Pasos de ejecucion

1. Definir el DTO de respuesta del dashboard.
2. Reusar progreso, correlativas y estado de materias como insumos.
3. Implementar servicio agregador del dashboard.
4. Exponer endpoint y documentar response.
5. Agregar pruebas del agregador.

## Comandos exactos

```bash
grep -R "Dashboard\|stats\|timeline\|available-subjects" -n src/main/java/aktech/planificador
./mvnw -q -DskipTests compile
./mvnw -q -DskipTests=false -Dtest=Subject*Test,CareerControllerTest test
```

```java
public record DashboardResponseDto(

        List<SubjectSummaryDto> recommendedSubjects,

}
```

## Verificacion

- Existe un endpoint claro para datos de dashboard.
- La respuesta agrega datos sin depender de DTOs legacy.
- Existen pruebas del servicio agregador o del endpoint.

## Resultado esperado

El backend queda listo para entregar una vista principal rica y estable cuando llegue la integracion del frontend.

## Criterio de cierre

- [ ] Existe DTO de dashboard.
- [ ] Existe servicio agregador.
- [ ] Existe endpoint documentado.
- [ ] La respuesta no depende de clases legacy.
- [ ] Hay pruebas asociadas.

## Troubleshooting breve

Si hace falta una referencia funcional, mirar el comportamiento legacy solo como insumo de negocio. No importar DTOs ni servicios legacy en el modulo nuevo.
