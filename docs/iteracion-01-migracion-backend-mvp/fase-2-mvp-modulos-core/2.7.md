# 2.7 Validaciones de Negocio

## Objetivo

Implementar en servicios las reglas de dominio del MVP para impedir transiciones invalidas, estados incoherentes y errores de consistencia academica.

## Por que se hace

Los controladores no deben decidir reglas de negocio. Si las validaciones quedan dispersas o incompletas, el backend acepta operaciones que contradicen la logica del producto.

## Pasos de ejecucion

1. Enumerar reglas del plan que deben bloquear operaciones.
2. Implementar excepciones claras y reutilizables.
3. Aplicar las validaciones en servicios transaccionales.
4. Cubrir con pruebas los casos que deben fallar y los que deben pasar.

## Comandos exactos

```bash
grep -R "validateCorrelativeRulesForStatus\|requiresResolvedCorrelatives\|validateNoCycle" -n src/main/java/aktech/planificador/modules
./mvnw -q -DskipTests compile

```

## Ejemplo de codigo

```java
if (SubjectStatus.CURSANDO.getValue().equalsIgnoreCase(normalizedStatus) && blocked) {
    throw new BusinessException("La materia esta bloqueada por correlativas pendientes y no puede pasar a cursando");
}
```

## Reglas cerradas en servicios

- SubjectService:
    - Bloquea create/update cuando el estado resultante es `cursando` y la materia sigue bloqueada por correlativas.
    - Bloquea create/update cuando el estado resultante es `aprobada` y existen correlativas pendientes.
    - La validacion vive en `validateCorrelativeRulesForStatus` y no en controladores.
- EquivalenceService:
    - Mantiene validacion de circularidad (`validateNoCycle`) en create y update.
    - Mantiene validacion de consistencia de carrera y pares duplicados directos/inversos.

## Verificacion

- No se puede aprobar materia sin correlativas resueltas.
- No se puede pasar a cursando una materia bloqueada.
- No se permiten equivalencias circulares.
- Las excepciones son claras y consistentes.
