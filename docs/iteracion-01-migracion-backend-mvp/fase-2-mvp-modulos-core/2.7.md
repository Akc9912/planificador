# 2.7 Validaciones de Negocio

## Objetivo

Implementar en servicios las reglas de dominio del MVP para impedir transiciones invalidas, estados incoherentes y errores de consistencia academica.

## Por que se hace

Los controladores no deben decidir reglas de negocio. Si las validaciones quedan dispersas o incompletas, el backend acepta operaciones que contradicen la logica del producto.

## Pasos de ejecucion

1. Enumerar reglas del plan que deben bloquear operaciones.
2. Implementar excepciones claras y reutilizables.
3. Aplicar las validaciones en servicios transaccionales.
4. Cubrir con pruebas los casos que deben fallar y los que deben pasar.

## Comandos exactos

```bash
grep -R "BusinessException\|NotFoundException\|validate\|blocked" -n src/main/java/aktech/planificador/modules src/main/java/aktech/planificador/shared
./mvnw -q -DskipTests compile
./mvnw -q -DskipTests=false -Dtest=CareerServiceTest,SubjectServiceTest,Equivalence*Test test
```

## Ejemplo de codigo

```java
if (subjectIsBlocked && newStatus == SubjectStatus.IN_PROGRESS) {
    throw new BusinessException("Blocked subject cannot move to IN_PROGRESS");
}
```

## Verificacion

- No se puede aprobar materia sin correlativas resueltas.
- No se puede pasar a cursando una materia bloqueada.
- No se permiten equivalencias circulares.
- Las excepciones son claras y consistentes.

## Resultado esperado

Las reglas clave del negocio quedan centralizadas en la capa correcta y el backend protege integridad funcional sin depender de chequeos del frontend.

## Criterio de cierre

- [ ] Las reglas estan en servicios.
- [ ] Los controladores no contienen logica de dominio.
- [ ] Existen excepciones compartidas o de modulo claras.
- [ ] Hay pruebas de rechazo y de aceptacion.

## Troubleshooting breve

Si una validacion se repite en varios servicios, extraerla a un helper de dominio o servicio dedicado. No copiar y pegar la misma regla en varios puntos.
