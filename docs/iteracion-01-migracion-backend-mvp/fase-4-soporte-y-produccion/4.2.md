# 4.2 Optimizacion de Performance

## Objetivo

Reducir el costo de consultas y listados del backend mediante indices, paginacion, optimizacion de queries y, si aporta valor real, cache en endpoints de solo lectura.

## Por que se hace

Un backend funcional pero lento traslada problemas a cada pantalla y complica la puesta en produccion. Esta fase convierte el MVP en una base operable bajo carga razonable.

## Pasos de ejecucion

1. Identificar consultas calientes y posibles N+1.
2. Revisar indices ya definidos en el schema.
3. Agregar paginacion en listados largos.
4. Optimizar repositorios con `JOIN FETCH`, `@EntityGraph` o consultas especificas cuando haga falta.
5. Evaluar cache solo en lecturas con beneficio claro.

## Comandos exactos

```bash
grep -n "CREATE INDEX" humanis_db_init.sql | head -n 80
grep -R "Pageable\|Cacheable\|@EntityGraph\|JOIN FETCH" -n src/main/java
./mvnw -q -DskipTests compile
./mvnw -q -DskipTests=false test
```

## Ejemplo de codigo

```java
public Page<SubjectResponseDto> getSubjects(UUID careerId, Pageable pageable) {
    return subjectRepository.findByCareerId(careerId, pageable).map(mapper::toResponse);
}
```

## Verificacion

- Existen indices para consultas sensibles o se documenta cuales faltan.
- Los listados largos usan paginacion.
- No hay N+1 evidentes en modulos clave.
- La suite sigue pasando luego de la optimizacion.

## Resultado esperado

El backend responde de forma mas eficiente y queda mejor preparado para una integracion real o una salida a produccion.

## Criterio de cierre

- [ ] Se revisaron indices y queries sensibles.
- [ ] Hay paginacion donde hace falta.
- [ ] Se redujeron consultas redundantes.
- [ ] La suite completa sigue en verde.

## Troubleshooting breve

No agregar cache como primera respuesta. Primero confirmar que el cuello de botella es real y que no se resuelve mejor con consultas e indices correctos.
