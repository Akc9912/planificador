# 4.1 Implementar Support Module

## Objetivo

Crear el modulo `support` para gestionar tickets de soporte del usuario, con tipos, estados y ownership correcto desde el backend.

## Por que se hace

El modulo de soporte cierra el set de dominios del backend y evita que reportes o sugerencias queden fuera de trazabilidad o mezclados con canales informales.

## Pasos de ejecucion

1. Crear estructura `modules/support`.
2. Modelar `support_tickets` con UUID, tipo y estado.
3. Implementar CRUD y filtros por estado y tipo.
4. Validar que cada usuario solo vea sus tickets.
5. Crear pruebas del modulo.

## Comandos exactos

```bash
find src/main/java/aktech/planificador/modules -path "*/support/*" | sort
grep -R "support_tickets\|Support\|Ticket" -n src/main/java/aktech/planificador humanis_db_init.sql
./mvnw -q -DskipTests compile
./mvnw -q -DskipTests=false -Dtest=Support*Test,ModuleBoundariesTest test
```

## Ejemplo de codigo

```java
@PostMapping("/api/support/tickets")
public ResponseEntity<SupportTicketResponseDto> createTicket(
        Authentication authentication,
        @RequestBody SupportTicketRequestDto request) {
    UUID userId = (UUID) authentication.getPrincipal();
    return ResponseEntity.status(HttpStatus.CREATED).body(supportService.create(userId, request));
}
```

## Verificacion

- Existe modulo `support` con capas completas.
- El usuario solo accede a sus tickets.
- Existen filtros por estado y tipo.
- Hay pruebas del modulo y guardrail modular.

## Resultado esperado

El backend puede registrar y consultar tickets de soporte como parte del producto, sin canales paralelos ni logica improvisada.

## Criterio de cierre

- [ ] Existe modulo de soporte completo.
- [ ] CRUD y filtros funcionan.
- [ ] Ownership validado.
- [ ] Pruebas del modulo en verde.

## Troubleshooting breve

No mezclar este modulo con auditoria. Soporte registra pedidos del usuario; auditoria registra acciones del sistema o del usuario sobre el sistema.
