# 4.5 Documentacion de Produccion

## Objetivo

Dejar el backend con documentacion final operativa: Swagger vigente, README util, variables de entorno claras y artefactos minimos para integracion y despliegue.

## Por que se hace

Un backend listo para produccion no se define solo por codigo. Tambien necesita instrucciones claras para levantarlo, integrarlo, verificarlo y operarlo.

## Pasos de ejecucion

1. Revisar configuracion OpenAPI y Swagger actual.
2. Confirmar que el README principal refleja el estado real del backend.
3. Documentar variables de entorno necesarias.
4. Generar o verificar artefactos de integracion como coleccion Postman si aplica.
5. Probar acceso local a `/swagger-ui.html` y `/v3/api-docs`.

## Comandos exactos

```bash
grep -R "OpenAPI\|swagger\|v3/api-docs\|swagger-ui" -n src/main/java README.md ARCHITECTURE.md MIGRATION_PLAN.md
./mvnw -q -DskipTests compile
./mvnw spring-boot:run
curl -i http://localhost:8081/v3/api-docs
curl -i http://localhost:8081/swagger-ui.html
```

## Ejemplo de codigo

```java
@Bean
public OpenAPI customOpenAPI() {
    return new OpenAPI().info(new Info().title("Planificador API").version("v1"));
}
```

## Verificacion

- Swagger y OpenAPI responden localmente.
- El README principal describe el estado real del backend.
- Las variables de entorno necesarias estan listadas.
- Existe documentacion suficiente para integracion y despliegue.

## Resultado esperado

El backend queda entendible y operable por un tercero sin depender del conocimiento informal del autor original.

## Criterio de cierre

- [ ] Swagger responde.
- [ ] El README principal esta alineado al estado real.
- [ ] Variables de entorno documentadas.
- [ ] Existe una guia minima de integracion y despliegue.

## Troubleshooting breve

Si Swagger no responde pero la app levanta, revisar primero `OpenApiConfig` y dependencias `springdoc` antes de asumir fallo general del backend.
