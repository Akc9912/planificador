# 3.5 Integraciones Entre Modulos

## Objetivo

Definir y validar las integraciones necesarias entre Event, Reminder, UserSettings, Audit y los modulos core usando solo eventos, interfaces compartidas y DTOs comunes.

## Por que se hace

Cuando aumentan los modulos, aumenta tambien la tentacion de resolver dependencias con imports directos. Esta tarea documenta el borde correcto entre dominios para que la arquitectura siga siendo modular.

## Pasos de ejecucion

1. Enumerar que datos necesita cada modulo de otro dominio.
2. Decidir si cada integracion es sincronica (`shared/api`) o asincronica (`shared/event`).
3. Crear o ajustar contratos compartidos minimos.
4. Implementar listeners o adaptadores necesarios.
5. Ejecutar el guardrail modular y las pruebas de integracion.

## Comandos exactos

```bash
find src/main/java/aktech/planificador/shared -type f | sort
grep -R "@EventListener\|ApplicationEventPublisher\|shared/api\|shared/event" -n src/main/java/aktech/planificador
./mvnw -q -DskipTests=false -Dtest=ModuleBoundariesTest test
./mvnw -q -DskipTests compile
```

## Ejemplo de codigo

```java
@EventListener
public void handleCareerDeleted(CareerDeletedEvent event) {
    subjectCleanupService.deleteByCareerId(event.careerId());
}
```

## Verificacion

- Cada integracion entre modulos esta documentada.
- Las dependencias sincroniccas usan interfaces y las asincronicas usan eventos.
- No hay imports directos entre modulos.
- `ModuleBoundariesTest` sigue pasando.

## Resultado esperado

Los modulos complementarios se conectan al resto del backend sin erosionar la arquitectura modular.

## Criterio de cierre

- [ ] Cada integracion tiene mecanismo definido.
- [ ] Los contratos compartidos son minimos y claros.
- [ ] No aparecen imports directos entre modulos.
- [ ] El guardrail modular sigue en verde.

## Troubleshooting breve

Si una integracion pide demasiados datos de otro modulo, probablemente el contrato esta mal definido. Reducirlo antes de codificar para no exportar medio dominio por error.
