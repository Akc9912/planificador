# 3.1 - Agregar constraints Jakarta en DTOs request

## Objetivo

Introducir validaciones declarativas en DTOs de entrada para estandarizar control de datos invalidos desde capa presentation.

## Razon Tecnica

Validar solo en servicios incrementa duplicacion y dificulta consistencia de errores. Jakarta Validation simplifica y unifica reglas de entrada.

## Comandos Exactos

```bash
rg -n "class .*RequestDto" src/main/java/aktech/planificador/modules/**/dto/*.java
rg -n "@NotBlank|@NotNull|@Size|@Pattern|@Min|@Max" src/main/java/aktech/planificador/modules/**/dto/*.java
./mvnw -DskipTests compile
```

## Ejemplo De Codigo (Si Aplica)

```txt
public class CareerCreateRequestDto {
  @NotBlank
  @Size(max = 150)
  private String name;
}
```

## Verificacion

1. DTOs de request relevantes contienen constraints de Jakarta.
2. Compilacion en verde tras agregar anotaciones.
3. No se rompen contratos existentes por cambios incompatibles.

## Resultado Esperado

Validacion declarativa activa en request DTOs de modulos del MVP.

## Troubleshooting Breve (Si Aplica)

- si hay validaciones duplicadas: mantener reglas criticas en application y entrada en DTO.
- si una regla depende de DB: no moverla al DTO; dejarla en application.
