# 3.3 - Estandarizar contrato JSON de errores API

## Objetivo

Unificar el formato de errores de la API en un contrato JSON estable para facilitar integracion frontend y observabilidad basica.

## Razon Tecnica

Respuestas de error como texto plano dificultan consumo consistente, testing y trazabilidad.

## Comandos Exactos

```bash
cat src/main/java/aktech/planificador/shared/exception/GlobalExceptionHandler.java
rg -n "ResponseEntity<String>|handle.*Exception" src/main/java/aktech/planificador/shared/exception/GlobalExceptionHandler.java
./mvnw -DskipTests=false -Dtest=*ControllerTest test
```

## Ejemplo De Codigo (Si Aplica)

```txt
{
  "timestamp": "2026-03-19T10:20:30Z",
  "status": 400,
  "error": "BAD_REQUEST",
  "message": "El campo nombre es obligatorio",
  "path": "/careers"
}
```

## Verificacion

1. GlobalExceptionHandler responde JSON uniforme para 400/401/403/404/409/500.
2. No se exponen stack traces ni detalles internos en respuesta.
3. Pruebas de controladores ajustadas y en verde.

## Resultado Esperado

Contrato de errores estable y reutilizable por clientes API.

## Troubleshooting Breve (Si Aplica)

- si cambia formato y rompe frontend: versionar contrato o adaptar parser del cliente.
- si hay mensajes inconsistentes: centralizar generacion de ApiErrorResponse.
