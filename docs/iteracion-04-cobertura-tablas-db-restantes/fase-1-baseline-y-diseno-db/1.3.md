# 1.3 - Definir reglas de ownership y validaciones criticas

## Objetivo

Documentar reglas de acceso por usuario y validaciones clave de negocio para las tablas faltantes.

## Razon Tecnica

Las politicas RLS existen en DB, pero backend debe reforzarlas para seguridad, UX y coherencia de errores.

## Comandos Exactos

```bash
rg -n "CREATE POLICY .*events|CREATE POLICY .*reminders|CREATE POLICY .*user_settings|CREATE POLICY .*audit_logs|CREATE POLICY .*support_tickets" humanis_db_init.sql
rg -n "ENABLE ROW LEVEL SECURITY" humanis_db_init.sql
```

## Ejemplo De Codigo (Si Aplica)

```txt
Reglas base:
- events/reminders/support_tickets/settings/audit_logs: solo owner
- event_schedules/event_subjects/reminder_subjects: ownership por parent
- validaciones: rango temporal valido, asociaciones unicas, estados permitidos
```

## Verificacion

1. Reglas de ownership estan documentadas por tabla.
2. Validaciones criticas estan definidas por agregado.
3. Se explicita estrategia de respuesta en errores de acceso/validacion.

## Resultado Esperado

Guia de seguridad y validacion lista para implementar sin contradicciones con DB.
