# 1.2 - Definir contratos API para modulos faltantes

## Objetivo

Definir contratos API minimos (endpoints y DTOs) para Events, Reminders, UserSettings, SupportTickets y AuditLogs.

## Razon Tecnica

API first reduce retrabajo entre backend y frontend y evita ambiguedad de alcance durante implementacion.

## Comandos Exactos

```bash
rg -n "@RequestMapping|@GetMapping|@PostMapping|@PutMapping|@DeleteMapping" src/main/java/aktech/planificador/modules/**/presentation/*.java
cat engineering/architecture/backend-architecture.md
cat engineering/architecture/java-backend-architecture-adaptation.md
```

## Ejemplo De Codigo (Si Aplica)

```txt
Contratos sugeridos:
- /events, /events/{id}, /events/{id}/schedules, /events/{id}/subjects
- /reminders, /reminders/{id}, /reminders/{id}/subjects
- /settings/me (GET, PUT)
- /support-tickets, /support-tickets/{id}
- /audit-logs (GET own)
```

## Verificacion

1. Cada tabla faltante tiene al menos un endpoint asociado en el diseno.
2. Se definen DTOs request/response iniciales por agregado.
3. El contrato respeta ownership por usuario autenticado.

## Resultado Esperado

Blueprint de API completo para ejecutar fases de implementacion sin vacios funcionales.
